# Events

The Slack bot can schedule events that trigger at specific times or immediately. Events are JSON files placed in the `events/` directory of the workspace. The bot watches this directory and executes events when they're due.

## Event types

| Type | Trigger | Persistence |
|------|---------|-------------|
| `immediate` | As soon as the file is created | Deleted after triggering |
| `one-shot` | At a specific date/time | Deleted after triggering |
| `periodic` | On a cron schedule | Persists until manually deleted |

## JSON format

**Immediate** — triggers as soon as the harness sees the file. Use for webhooks, external signals, or programs that need to notify a channel right away.

```json
{"type": "immediate", "channelId": "C123ABC", "text": "New GitHub issue opened"}
```

**One-shot** — triggers once at a specific date and time, then the file is deleted. Use for reminders and scheduled tasks.

```json
{"type": "one-shot", "channelId": "C123ABC", "text": "Remind about standup", "at": "2025-12-15T09:00:00+01:00"}
```

**Periodic** — triggers repeatedly on a cron schedule. The file persists until manually deleted. The optional `threadTs` field targets a specific Slack thread instead of posting to the channel root.

```json
{"type": "periodic", "channelId": "C123ABC", "text": "Check inbox", "schedule": "0 9 * * 1-5", "timezone": "Europe/Vienna", "threadTs": "1234567890.123456"}
```

All event types accept an optional `threadTs` field to post the response into an existing Slack thread rather than starting a new one.

## Timezones

Timezone handling differs by event type:

- **One-shot events**: include the offset directly in the `at` field using ISO 8601 format (e.g., `2025-12-15T09:00:00+01:00`, `2025-12-15T09:00:00-05:00`)
- **Periodic events**: set the `timezone` field to an IANA timezone name (e.g., `America/New_York`, `Europe/Vienna`)

Cron format for the `schedule` field is `minute hour day-of-month month day-of-week`. Common examples:

- `0 9 * * *` — daily at 9:00
- `0 9 * * 1-5` — weekdays at 9:00
- `30 14 * * 1` — Mondays at 14:30
- `*/15 * * * *` — every 15 minutes

## Silent completion

For periodic events that check for activity (inbox, notifications, etc.), the bot may find nothing to report. The agent can respond with `[SILENT]` to suppress the status message entirely. This prevents channel spam from recurring checks that have nothing to say.

Example: a periodic event checks for new emails every 15 minutes. If there are no new emails, the agent responds `[SILENT]` and nothing is posted. If there are new emails, the agent posts a summary.

## Limits

- Maximum 5 events can be queued per channel at a time
- Use unique filenames (e.g., `reminder-$(date +%s).json`) to avoid overwriting existing events
- Periodic events should debounce — check an inbox every 15 minutes rather than creating one event per incoming item

## Events vs Heartbeats

> Events and [Heartbeats](/open-harness/guide/heartbeats) are different scheduling systems. Events are Slack-bot-specific -- JSON files that target a Slack channel. Heartbeats are sandbox-level -- cron schedules that invoke an agent CLI on a `.md` file. Use events for Slack-channel tasks; use heartbeats for general sandbox automation.
